Sentro — Error Tracking &
Agent Observability

Error tracking and observability built for AI agents. Sentro gives you deep visibility into every run, step, tool call, and LLM call your agents make — plus traditional error tracking for the rest of your stack.

How Sentro differs from Sentry Sentry groups errors by stack trace and release. Sentro does that too — and adds an agent-first hierarchy: every error, LLM call, and tool call can be traced back to its parent Run → Step → ToolCall / LlmCall, so you can replay exactly what your agent did before it failed.
🐛

Error Tracking

Automatic fingerprinting, stack trace capture, status management, and affected-run linking.

🤖

Agent Run Tracing

Track every agent run end-to-end with goal, model, trigger, and step-by-step timeline.

🔧

Tool Call Monitoring

Latency, input/output, error rates — broken down per tool for every step.

🧠

LLM Call Tracking

Model, provider, token counts, latency, and cost captured for each inference call.

💰

Cost Tracking

Aggregate LLM spend by model, run, and time range. Set threshold alerts before bills surprise you.

🔔

Alerting

Webhook alerts for error spikes, agent failure rate, and cost threshold breaches.

Quick Start

Prerequisites

  • Docker 24+ and Docker Compose v2+
  • A terminal and a browser — that's it

Start Sentro in three steps

  1. Clone the repository
    bash
    git clone https://github.com/your-org/sentro.git
cd sentro
  2. Start services with Docker Compose
    bash
    docker compose up -d

    This starts the Next.js application on port 3000 and a Postgres 16 database. First boot takes ~30 s while it builds the image.

  3. Open your browser

    Navigate to http://localhost:3000. You'll be redirected to setup to create your admin account.

First-time setup

  1. Fill in your email and a strong password on the Setup page — this creates the first admin account. Setup is only available once; if you need to reset, clear the users table.
  2. Log in and click New Project in the sidebar.
  3. Give your project a name (e.g. my-agent) and click Create.
  4. Copy the DSN shown on the project page — it looks like:
    http://<token>@localhost:3000/api/ingest/<projectId>
Production note Set SESSION_SECRET to a random 32-byte hex string before exposing Sentro to the internet. The default value is only safe for local development.

SDK Installation & Setup

Install

bash
npm install @sentro/sdk
# or
pnpm add @sentro/sdk
yarn add @sentro/sdk

Initialize

Create a single Sentro instance and export it — typically in lib/sentro.ts:

typescript
import { Sentro } from "@sentro/sdk";

export const sentro = new Sentro({
  dsn: "http://abc123@localhost:3000/api/ingest/proj_xyz",
  capturePrompts: false,  // set true to store prompt/response bodies
  flushIntervalMs: 1000,  // batch send every 1 s
  maxBatchSize: 100,       // flush immediately when buffer hits 100
  defaultTags: {
    env: "production",
    version: "1.4.2",
  },
});

Configuration options

OptionTypeDefaultDescription
dsnstringrequiredFull DSN URL from your project settings.
capturePromptsbooleanfalseWhen true, prompt messages and LLM responses are stored verbatim. Keep false if prompts contain PII.
flushIntervalMsnumber1000How often (in ms) the SDK flushes its internal buffer to the ingest endpoint.
maxBatchSizenumber100Maximum number of events per batch. Flush happens immediately when this is reached.
defaultTagsRecord<string,string>—Tags merged into every event sent from this client instance.
Graceful shutdown Call await sentro.shutdown() before your process exits to ensure all buffered events are flushed. In serverless environments use await sentro.flush() at the end of each function invocation.

SDK Guide — Error Tracking

Capture exceptions

Pass any Error object to captureException. Sentro records the message, stack trace, and any active tags/context.

typescript
import { sentro } from "./lib/sentro";

try {
  await processOrder(orderId);
} catch (err) {
  sentro.captureException(err instanceof Error ? err : new Error(String(err)));
  throw err;
}

Capture messages

Send a plain string at a specific severity level: "error", "warning", "info", or "debug".

typescript
sentro.captureMessage("Payment gateway timed out after 3 retries", "warning");
sentro.captureMessage("Nightly sync completed successfully", "info");

Tags and context

Tags are indexed string key/value pairs — useful for filtering in the dashboard. Context stores arbitrary structured data shown alongside each event.

typescript
// Tags become filter dimensions in the Issues view
sentro.setTags({
  userId: user.id,
  plan: "enterprise",
  region: "us-east-1",
});

// Context stores any JSON — shown in the event detail drawer
sentro.setContext({
  request: { method: "POST", url: "/api/checkout", ip: req.ip },
  order: { id: orderId, total: cart.total },
});

Express global error handler

typescript
import express from "express";
import { sentro } from "./lib/sentro";

const app = express();

// … your routes …

// Must be last middleware — 4 args = error handler
app.use((err: Error, req, res, next) => {
  sentro.setContext({
    request: {
      method: req.method,
      url: req.originalUrl,
      body: req.body,
    },
  });
  sentro.captureException(err);
  res.status(500).json({ error: "Internal server error" });
});

app.listen(3001);
How grouping works Sentro fingerprints each event by hashing the error type and the first non-library frame of the stack trace. All events with the same fingerprint are merged into a single Issue in the dashboard. Identical errors fired 1000 times show as one issue with a count of 1000.

SDK Guide — Agent Observability AI

Sentro models agent execution as a four-level hierarchy:

Run
→
Step
→
ToolCall
One agent invocation
Reasoning turn
External action
LlmCall
also under each Step

Explicit API

Full manual control — start and end each entity yourself:

typescript
import { sentro } from "./lib/sentro";

async function runResearchAgent(query: string) {

  // 1. Start a run
  const run = sentro.startRun({
    agent:   "research-agent",
    goal:    query,
    model:   "gpt-4o",
    trigger: "api",
    metadata: { userId: "u_123" },
  });

  try {
    // 2. Create a step (reasoning turn)
    const step1 = run.step("Search for recent papers");

    // 3. Record an LLM call
    const llm1 = step1.llmCall({
      model:       "gpt-4o",
      provider:    "openai",
      temperature: 0.2,
      messages: [
        { role: "system", content: "You are a research assistant." },
        { role: "user",   content: query },
      ],
    });

    const completion = await openai.chat(...);

    await llm1.end({
      promptTokens:     completion.usage.prompt_tokens,
      completionTokens: completion.usage.completion_tokens,
      cost:             0.0042,   // compute from token pricing
      response:         completion.choices[0].message,
    });

    // 4. Record a tool call
    const tc = step1.toolCall("web_search", {
      input: { query: "LLM agent evaluation 2025", num_results: 10 },
    });

    const results = await webSearch(query);
    await tc.end({ result: results.slice(0, 3) });

    // 5. End the step
    await step1.end();

    // 6. End the run
    await run.end({ status: "success" });
    return results;

  } catch (err) {
    await run.error(err instanceof Error ? err : new Error(String(err)));
    throw err;
  }
}

Wrapper API

The wrapper API uses callbacks to automatically handle success/failure and call .end() for you. Use it when you want less boilerplate:

typescript
// sentro.trace() — wraps an entire run
const result = await sentro.trace(
  "research-agent",
  { goal: "Summarise AI news for this week", model: "claude-3-5-sonnet" },
  async (run) => {

    // run.trace() — wraps a single step
    const headlines = await run.trace("Fetch top headlines", async (step) => {

      // step.traceToolCall() — wraps a tool call
      return step.traceToolCall(
        "rss_fetch",
        { url: "https://techcrunch.com/feed", limit: 20 },
        async (_tc) => {
          return fetchRss("https://techcrunch.com/feed");
        }
      );
    });

    return { headlines };
  }
);

When to use which

APIBest for
ExplicitLoops, conditional branching, retry logic, or when you need the run/step handle to pass to other functions.
Wrapper (trace)Simple linear flows where you just want automatic start/end and error capture with minimal code.

Real-world multi-step agent example

typescript
import OpenAI from "openai";
import { sentro } from "./lib/sentro";

const openai = new OpenAI();

/**
 * An agent that answers a question by searching the web,
 * reading each page, and synthesising a final answer.
 */
export async function answerAgent(question: string): Promise<string> {
  return sentro.trace(
    "answer-agent",
    { goal: question, model: "gpt-4o-mini", trigger: "api" },
    async (run) => {

      // Step 1: Decide what to search for
      const searchQuery = await run.trace("Plan search queries", async (step) => {
        const llm = step.llmCall({ model: "gpt-4o-mini", provider: "openai" });
        const res = await openai.chat.completions.create({
          model: "gpt-4o-mini",
          messages: [{ role: "user", content: `Turn into a web search query: ${question}` }],
        });
        await llm.end({
          promptTokens:     res.usage?.prompt_tokens,
          completionTokens: res.usage?.completion_tokens,
        });
        return res.choices[0].message.content ?? question;
      });

      // Step 2: Search + read pages
      const pages = await run.trace("Search and read sources", async (step) => {
        const hits = await step.traceToolCall(
          "web_search", { q: searchQuery },
          async () => webSearch(searchQuery)
        );
        return Promise.all(
          hits.slice(0, 3).map((url) =>
            step.traceToolCall("read_page", { url }, async () => readPage(url))
          )
        );
      });

      // Step 3: Synthesise final answer
      return run.trace("Synthesise answer", async (step) => {
        const llm = step.llmCall({ model: "gpt-4o", provider: "openai" });
        const final = await openai.chat.completions.create({
          model: "gpt-4o",
          messages: [{
            role: "user",
            content: `Answer ${question} using: ${pages.join("\n\n")}`,
          }],
        });
        await llm.end({
          promptTokens:     final.usage?.prompt_tokens,
          completionTokens: final.usage?.completion_tokens,
          cost: 0.021,
        });
        return final.choices[0].message.content ?? "";
      });
    }
  );
}

Dashboard Guide

Issues page

The Issues page lists all error groups for a project. Sentro fingerprints each incoming event by hashing the error type and stack frame — identical errors are merged into one issue rather than creating thousands of individual records.

  • Status tabs — Open, Resolved, Ignored. Resolving an issue keeps it in the list; if the same fingerprint fires again it automatically re-opens.
  • Affected runs badge — shows how many distinct agent runs this issue appeared in. Useful for assessing blast radius on agentic systems.
  • Event count — total occurrences since first seen.
  • Last seen — issues are sorted by recency by default so your noisiest current problem is always first.

Agent Runs page AI

A filterable list of every agent run sent to this project.

  • Stats bar — at the top, summarises the filtered window: total runs, success rate %, average duration, total tokens, and total cost.
  • Filter by agent — if you run multiple agents in one project, the dropdown lets you isolate one.
  • Filter by status — running / success / failure / timeout.
  • Step count — how many reasoning steps this run went through before finishing.
  • Cost / Tokens — per-run totals aggregated from all LLM calls within.

Run Detail & Step Replay AI

Click any run to open its timeline. Each step is shown in sequence order with its content, start/end time, and duration. Expand any step to see:

  • Tool calls — tool name, input parameters, output, status (success / error), and latency in ms.
  • LLM calls — model, provider, token counts (prompt + completion), cost, and latency.
  • Recent errors — up to 10 events that were captured during this run, linked to their issue.

The timeline is read left-to-right, top-to-bottom in chronological order. You can replay the exact sequence of decisions and actions the agent took.

Performance page Cost

Aggregate performance metrics for the selected time window (default: last 24 hours).

  • LLM cost by model — a table showing each model, call count, total tokens, total cost, and average latency. Use this to spot expensive models or prompts.
  • Slowest tool calls — top 20 tool calls by latency in the window. Links to the parent run for context.
  • Slowest LLM calls — same for inference calls. Useful for identifying unusually long completions.

Alerts page

Configure webhook alerts that fire when thresholds are breached. Alerts are evaluated every 60 seconds by the background worker.

  • error_spike — fires when the number of new errors in the last N minutes exceeds a threshold. Config keys: windowMinutes, threshold.
  • failure_rate — fires when the fraction of failed agent runs exceeds a percentage. Config keys: windowMinutes, thresholdPct.
  • cost_threshold — fires when total LLM spend in the window exceeds a dollar amount. Config keys: windowMinutes, thresholdUsd.

Webhooks receive a JSON POST with fields: ruleId, ruleName, type, projectId, triggeredAt, and payload (metric details).

API Reference

Authentication

Sentro has two authentication surfaces:

  • Ingest API (POST /api/ingest) — authenticated by DSN. The SDK sends the token as a Authorization: Bearer <token> header. You can also embed it in the payload's dsn field.
  • Dashboard API (all other routes) — authenticated by session cookie. Log in via the UI or POST to /api/auth/login to obtain a session cookie.

Ingest endpoint

POST /api/ingest

Accepts a batch of typed events from the SDK. The SDK handles this automatically — you only need this reference for custom integrations.

json
{
  "dsn":   "<projectId-or-dsnToken>",
  "batch": [
    {
      "type":       "event",        // event | run.start | run.end | step.start
      "timestamp":  "2025-04-12T10:00:00.000Z",
      "level":      "error",
      "message":    "Cannot read properties of undefined",
      "stackTrace": "Error: Cannot read...\n  at ...",
      "tags":       { "env": "production" },
      "context":    {}
    }
  ]
}

Event types:

typeRequired fieldsDescription
eventlevel, messagePlain error or message event.
run.startrunId, agentBegins an agent run.
run.endrunId, statusEnds a run. status: success | failure | timeout.
step.startrunId, stepId, seq, contentBegins a step in a run.
step.endrunId, stepId, seqEnds a step.
tool_call.startrunId, stepId, toolCallId, toolNameBegins a tool call.
tool_call.endrunId, stepId, toolCallId, toolNameEnds a tool call with optional result/error.
llm_call.startrunId, stepId, llmCallId, modelBegins an LLM inference call.
llm_call.endrunId, stepId, llmCallIdEnds LLM call with token counts and cost.

Response: 202 Accepted — { "accepted": 3 }
Rate limit: configurable per-project (default 1000 events/min). Returns 429 when exceeded.

Configuration

Environment variables

VariableRequiredDescription
DATABASE_URLYesPostgreSQL connection string. Example: postgresql://sentro:sentro@db:5432/sentro
SESSION_SECRETYesSecret key for signing session tokens. Use a random 32-byte hex string in production.
NEXT_PUBLIC_APP_URLYesPublicly reachable base URL of the Sentro instance. Used to construct DSN URLs. Example: https://sentro.example.com
Security Never commit a .env file with real credentials. Use Docker secrets or your hosting platform's secret manager in production. The default SESSION_SECRET of change-me-in-production must be changed before any public exposure.

Per-project settings

Each project has two configurable properties (editable via the Settings tab or directly in Postgres):

FieldDefaultDescription
retentionDays30Events and runs older than this are purged by the nightly cleanup job.
rateLimitPerMinute1000Maximum ingest events accepted per minute per DSN. Requests over the limit receive a 429 response.

Self-Hosting Guide

Docker Compose (recommended)

The included docker-compose.yml starts everything you need:

yaml
version: "3.8"
services:
  app:
    build: .
    ports:
      - "3000:3000"
    environment:
      DATABASE_URL: "postgresql://sentro:sentro@db:5432/sentro?schema=public"
      SESSION_SECRET: "${SESSION_SECRET:-change-me-in-production}"
      NEXT_PUBLIC_APP_URL: "http://localhost:3000"
    depends_on:
      db:
        condition: service_healthy
    restart: unless-stopped

  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER:     sentro
      POSTGRES_PASSWORD: sentro
      POSTGRES_DB:       sentro
    volumes:
      - sentro-data:/var/lib/postgresql/data
    healthcheck:
      test:     ["CMD-SHELL", "pg_isready -U sentro"]
      interval: 5s
      timeout:  5s
      retries:  5
    restart: unless-stopped

volumes:
  sentro-data:

Data persistence

All data is stored in the sentro-data named Docker volume mapped to /var/lib/postgresql/data. The volume persists across docker compose down restarts. To fully wipe it run:

bash
docker compose down -v   # removes volumes — ALL data is lost

Backup strategy

bash
# Dump to a compressed file
docker exec sentro-db-1 pg_dump -U sentro sentro \
  | gzip > sentro-backup-$(date +%Y%m%d).sql.gz

# Restore
gunzip -c sentro-backup-20250412.sql.gz \
  | docker exec -i sentro-db-1 psql -U sentro sentro

Architecture

Data flow

SDK
@sentro/sdk
→
POST /api/ingest
auth + rate limit
→
In-Memory Buffer
lib/ingestion/buffer.ts
→
Processor
lib/ingestion/processor.ts
→
PostgreSQL
Prisma ORM
↑ Dashboard reads directly from Postgres via Next.js API routes

The ingest endpoint is intentionally synchronous for simplicity — the buffer flush happens in a setInterval within the Next.js process using pg-boss's job queue, which keeps transaction ordering consistent without an external broker.

Background jobs

Two recurring jobs run inside the application process via pg-boss (a Postgres-backed job queue):

  • alert-check — runs every minute (* * * * *). Evaluates all enabled alert rules, computes current metric values, and fires webhooks for rules that breach their threshold.
  • cleanup — runs daily at 3:00 AM (0 3 * * *). Deletes events, runs, steps, tool calls, and LLM calls that are older than each project's retentionDays setting.

Database schema overview

The schema has four main entity clusters:

Auth

User ↔ Session — standard credential + session token table. Sessions expire and are validated on every dashboard API call.

Projects & Error Tracking

Project → EventGroup → Event. Each inbound event type is fingerprinted and upserted into its parent EventGroup. Groups have a status (open/resolved/ignored) and counters updated atomically on each event.

Agent Observability

AgentRun → Step → ToolCall / LlmCall. Runs accumulate totalTokens and totalCost as LLM calls finish. Steps carry a sequenceNumber for ordered replay. Tool calls and LLM calls store both startedAt and latency for performance queries.

Alerting

AlertRule → AlertHistory. Each rule stores its config as JSON so new rule types can be added without schema changes. History records the webhook HTTP status for debugging delivery failures.

Key indexes

Performance-sensitive indexes beyond primary keys:

  • EventGroup(projectId, status, lastSeen DESC) — powers the issues list.
  • AgentRun(projectId, startedAt DESC) and (projectId, agentName, startedAt DESC) — powers the runs list and per-agent filter.
  • ToolCall(projectId, latencyMs DESC) and LlmCall(projectId, latencyMs DESC) — powers the performance page.
  • LlmCall(projectId, model, startedAt DESC) — powers cost-by-model aggregation.

Roadmap

Sentro is open-source and actively developed. Here's what's planned:

v1.1

Framework integrations & guardrails

  • Python SDK (pip install sentro-sdk)
  • LangChain callback handler — automatic tracing with zero manual instrumentation
  • CrewAI integration — trace crew tasks and agent interactions
  • Vercel AI SDK middleware — wrap generateText / streamText calls automatically
  • Drift & guardrail alerts — fire when agent behaviour deviates from a baseline (token count, step count, tool call frequency)
  • Session replay UI — visualise full run timelines as an interactive Gantt-style chart
v1.5

Redis & horizontal scaling

  • Optional Redis adapter for session store
  • Redis-backed job queue replacing pg-boss for better horizontal scaling
  • Multiple app replicas behind a load balancer
  • Configurable event buffer with at-least-once delivery guarantees
v2

Separate ingestion service & ClickHouse

  • Dedicated lightweight ingest microservice (unbundled from the Next.js app)
  • ClickHouse for analytical queries — real-time aggregations over billions of events
  • Postgres read replica for dashboard queries
  • Kafka/Redpanda as the event bus between ingest and storage
v3

SaaS mode & multi-tenancy

  • Organisation-level isolation with role-based access (admin / member / viewer)
  • Usage metering and billing integration
  • Managed Postgres per organisation option
  • SSO / SAML support
  • Audit log for compliance
